Object-Oriented Design: combines data and methods together into a cohesive whole (object)

Object-Oriented vs Procedural
  • Example: Get salt
    • OO: "Please pass me the salt"
    • Procedural: "Please move your right hand to the left until it contacts the salt shaker, grasp it, lift it from the table, move it in a natural arc in my direction, stop when it contacts my hand, wait until my hand has closed around it, then release it"
  • OO way:    
    • leave it up to the object to carry out the request any way it chooses
    • Objects are responsible for their own actions

Concepts
  • Encapsulation:  
    • Each object is a "black box" which contains code, data, and methods that can be invoked
    • grouping of related ideas into one unit which we can refer to by a single name
      • e.g. fields, methods, classes
    • providing access to an object only through invoking its method, while keeping the details
    • Information hiding: restrict external visibility of an object's information
  • Class:  
    • a blue print   or specification from which objects are created
      • define the structure and services of a set of objects
      • define which parts of an object can be accessed from outside
      • implement operations (method)
    • a Class is a type, an object is an instance
  • Object:  
    • Object is a run time concept
    • an object is only created during program execution
  • Access:
    • public: accessible from outside
    • private: accessible only from within the class
    • protected: accessible from within the class and from within subclasses
  • Inheritance:
    • If class B inherits information from class A, it implicitly acquires the attributes and methods of A
      • B reuses attributes and methods of A
      • A is the superclass, B is the subclass
      • "is-a" relationship: an object of type B "IS A" object of type A
      • B can also have new attributes in addition to the derived parts
    • Inheritance relationship between classes forms a class hierarchy
  • Polymorphism:
    • An object can be of several different types


Evaluation of OOD
  • Coupling: (three types)
    • interaction:  
      • when the methods of a class invoke methods of another class
        • a method of class A should NOT directly manipulate the attributes of another class B
    • component:  
      • when a class A has variables of another class C
        • A has instance variables of type C
        • A has a method with a parameter of type C
        • A has a method with a local variable of type C
      • when A is coupled with C, it is coupled with all subclasses of C as well
    • inheritance:
      • two classes are coupled if one is a subclass of the other
        • subclass should only add features (attributes, methods) to its superclass, as opposed to modifying the features it inherits
  • Cohesion
    • an intramodule concept that focuses on why elements are together
      • only elements tightly related should exist together in a class
      • higher cohesion -> lower coupling as many otherwise interacting elements are already contained in the module
    • three types of cohesion
      • method cohesion: all elements within a method help to implement a clearly specified function
      • class cohesion: a class itself should be cohesive with each of its methods and attributes contributing to implement the class's clearly specified role
      • inheritance cohesion: focuses on why classes are together in a hierarchy
        • generalization-specialization: good
        • reuse because a pre-existing class does most of what you need: avoid
  • Open-closed principle
    • classes should be open for extension but closed for modification
      • behavior can be extended to accommodate new requirements, but existing code is not modified
    • satisfied  by using inheritance and polymorphism
      • allow creating a new class to extend behavior without changing the original class

Unified Model Language (UML) and Modeling
  • UML is a graphical design notation useful for OO analysis and design
    • It contains notations to model both structural and behavioral aspects of these systems
      • structure-related notations
        • class, object, package, use case, component
      • behavior-related notations
        • structure, collaboration, state, and activity diagrams
    • Class diagrams:
      • specify classes in a system, the association between classes, inheritance hierarchy
      • represent static structures
    • Interaction diagrams:
      • provide insight into a system's dynamic behavior
      • Sequence diagram
        • objects participating in an interaction are shown at the top
        • for each object, a vertical bar represents its lifeline
        • a message from one object to another is represented as a labeled arrow
        • the ordering or messages is captured along vertical axis

      • Collaboration diagram
        • Instead of timeline, shows the instantiation of associations between classes at run-time
        • the ordering of a set of messages is captured by numbering them

OO Design Methodologies
  • goal: identify classes, understand their behavior, and relationships
  • Steps:
  1. Analyze use cases
  2. Create activity diagrams for each use case
  3. Create class diagram based on 1 and 2
  4. Create interaction diagrams for activities contained in diagrams created in step 2
  5. Create state diagrams for classes created in step 3
  6. Iterate, each step above will reveal information about the other models that will need to be updated



Association between Objects
  • Dependency Association: An object's operation requires another type of object
    • e.g. a Driver object requires a Car object to drive
  • Composition Association: An object is composed of other type objects, implies ownership
    • e.g. a Vehicle object owns a Engine object
    • If the owner object no longer exist, its containing object also disappear
  • Aggregation Association: An object involves other types of object, but does not imply ownership
    • e.g. a Course object involves a number of Student objects
    • If the course no longer exist, student objects are not affected

SOLID Principle
  • Single Responsibility Principle
    • A class should have a single responsibility and only have its primary behaviors
    • It should only need to know things it requires
  • Open Closed Principle
    • software modules should be open for extension and closed for modification
    • we shouldn't modify internal structure of existing classes in order to extend more functionalities
  • Liskov Substitution Principle
    • Subtypes (subclass) must be substitutable for their basetypes (superclass)
    • a subclass must have an "is-a" relationship with its superclass
    • a Subclass should not force changes in its superclass
  • Interface Segregation Principle
    • no client should be forced to depend on methods that they don't use
    • usually occur when the class that is depended on has too many  responsibility
      • The dependent class will need to recompile every time the fat class changes, even if it is not affected by the change
    • use interface to separate behaviors to prevent tight coupling
      • reference the object using the interface variable to only access methods required
  • Dependency Inversion Principle
    • higher level modules should not depend on lower level (which are susceptible to change)
    • Both should depend on Abstractions
      • because abstract classes and interfaces don't change as often as concrete derivatives


Dependency Injection
  • for separating the construction of an object from its eventual use
    • object should not be responsible for instantiating each other, leads to tight coupling
  • instead of hard coding of a particular implementation, make it more generic by injecting an object in the constructor
  • can use a framework to handle the dependency management, which contains rules and instructions  for how to correct an instance properly
    • e.g. Spring
